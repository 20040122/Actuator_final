## 一、引言

随着对天基任务协同、自主决策执行和多平台联合控制需求的不断提升，传统以单节点、单任务脚本为核心的任务执行方式，已经难以满足复杂任务场景下对灵活编排、状态感知和协同控制的要求。在多星协同任务中，系统不仅需要描述“哪一颗卫星执行哪一项任务”，还需要明确“任务在执行过程中如何决策、如何共享信息、如何根据环境变化调整后续动作”。因此，将调度计划、任务执行逻辑、节点通信机制和运行时状态管理进行解耦，构建一个结构清晰、职责明确、便于扩展的执行框架，具有较强的工程意义和研究价值。

本项目正是在这一背景下构建的一个基于 C++11 的多星任务执行仿真系统。系统以多卫星任务协同执行为核心对象，以调度文件和行为树为主要输入，通过统一的协调器组织任务分发，通过卫星模拟器承载节点侧执行逻辑，通过执行器解释和运行行为树定义，从而形成一条完整的“任务规划输入 - 行为解析 - 指令执行 - 状态回传”闭环流程。项目当前已经能够完成从调度文件加载任务、从行为库中解析任务模板、向各卫星节点分发任务、在节点侧执行行为树并回传执行结果的基本过程，具备较完整的原型系统结构。

从实现方式上看，当前工程并非一个严格意义上的真实分布式部署系统，而是一个强调架构表达和功能验证的单进程多节点仿真框架。协调器 `Coordinator` 与多个卫星模拟器 `SatelliteSimulator` 运行在同一个进程内部，节点之间的消息通信统一抽象为 `InterSatComm` 模块，并优先采用本地消息处理器进行投递；行为树执行则由 `GenericExecutor` 负责，支持条件判断、顺序执行、选择执行、并行节点语义以及一组面向任务场景的内置命令；星间信息共享当前通过 `SEND` / `RECEIVE` 命令结合进程内共享邮箱机制进行模拟。这种设计方式虽然简化了真实部署环境中的网络复杂性，但完整保留了任务调度、消息协议、行为执行和状态管理等核心机制，使系统非常适合作为算法验证、流程演示和后续架构扩展的基础平台。

从工程角度分析，本项目的价值主要体现在以下几个方面。首先，它把“调度计划”与“任务执行逻辑”分离，调度文件只负责描述任务分配关系，而具体执行过程由行为树定义，使系统具有更好的配置化和可维护性。其次，它通过行为树模型描述任务步骤、条件分支与控制流，使复杂任务流程能够以结构化方式表达，避免将大量规则硬编码在程序逻辑中。再次，它通过协调器、通信抽象和节点模拟器构建了一个相对完整的协同执行框架，使得系统不仅能够执行单星任务，也能够表达多星之间的信息交互与结果汇总。最后，项目已经预留了继续向真实通信、多线程并发、复杂任务控制和自动化测试体系演进的空间，具备较好的扩展基础。

本文档的编写目的，是对 `src` 目录下的源码实现进行系统说明，帮助读者从整体架构、模块职责、核心类关系、运行流程和关键数据结构几个维度理解项目代码。文档内容不侧重算法推导，而侧重工程实现层面的源码解读，重点回答以下问题：系统由哪些模块组成，各模块分别承担什么职责；任务是如何从输入文件进入系统并最终被执行的；协调器、执行器、解析器和通信模块之间如何协同工作；当前实现具备哪些能力，又存在哪些明显的扩展方向。通过阅读本文档，开发人员可以更高效地理解项目结构，定位功能入口，并为后续二次开发、功能增强和文档整理打下基础。

## 二、`src` 目录结构说明

`src` 目录按职责划分为以下几个子模块：

| 目录 | 作用 |
| --- | --- |
| `src/main.cpp` | 程序入口，创建协调器并启动系统 |
| `src/core` | 公共类型、时间工具、日志工具 |
| `src/parser` | 调度文件和行为库解析，行为模板实例化 |
| `src/constraint` | 条件表达式求值 |
| `src/executor` | 行为树执行器、变量管理器、卫星模拟器 |
| `src/coordinator` | 协调器、节点注册、通信协议与消息路由 |
| `src/input` | 示例输入文件，包括调度计划和行为树库 |
| `src/third_party/nlohmann` | 第三方 JSON 库，不建议在本地改动 |

可从源码阅读角度把系统理解为下面这条调用链：

```text
main
  -> Coordinator
     -> ScheduleParser
     -> NodeRegistry
     -> InterSatComm
     -> SatelliteSimulator
        -> GenericExecutor
           -> BehaviorLibraryParser
           -> BehaviorTreeParser
           -> VariableManager
           -> ConstraintEvaluator
```

## 三、程序主流程

### 3.1 启动入口

程序入口位于 `src/main.cpp`。其主要逻辑如下：

1. 注册 `SIGINT` 和 `SIGTERM` 信号处理函数。
2. 创建默认 `CoordinatorConfig`，并将协调器节点 ID 设置为 `COORD_MAIN`。
3. 构造 `Coordinator` 实例。
4. 调用 `initialize("src/input/schedule.json")` 加载调度文件并初始化系统。
5. 调用 `run()` 开始任务分发和执行。
6. 调用 `shutdown()` 做统一资源释放。

这说明主函数本身很薄，真正的系统组织逻辑都集中在 `Coordinator` 中。

### 3.2 协调器初始化流程

`Coordinator::initialize()` 是系统启动的核心阶段，主要完成以下工作：

1. 加载调度文件，得到多颗卫星及其任务列表。
2. 初始化 `NodeRegistry` 节点注册表。
3. 根据调度结果注册所有卫星节点。
4. 初始化 `InterSatComm` 通信模块。
5. 为每颗卫星创建一个 `SatelliteSimulator`。
6. 初始化消息路由表。
7. 为协调器和每颗卫星注册本地消息处理器。

完成后，系统已经具备“可分发任务、可接收回执、可执行行为树”的基本能力。

### 3.3 任务执行流程

`Coordinator::run()` 的核心逻辑可以概括为：

1. 启动通信模块。
2. 根据调度结果为每颗卫星初始化任务跟踪状态。
3. 调用 `distributeAllTasks()` 按卫星批量下发任务。
4. 等待各卫星返回 `BATCH_TASK_ASSIGN_ACK`。
5. 在 60 秒超时内统计执行结果。
6. 输出成功/失败卫星数量并结束运行。

当前实现里，协调器重点关注“整颗卫星是否完成其任务批次”，而不是逐条维护复杂的任务状态机。

## 四、核心模块详解

为便于从源码层面理解项目，本章不只说明“模块做什么”，还进一步说明“模块内部有哪些关键组件”“这些组件之间如何协作”“它们的输入输出分别是什么”。从当前代码结构看，系统的组件可以分为五类：

- 基础支撑组件：提供公共类型、时间函数和日志能力。
- 解析组件：负责把 JSON 输入转为内存对象，并把行为模板实例化。
- 条件计算组件：负责布尔约束求值。
- 执行组件：负责变量管理、行为树解释执行和卫星侧任务处理。
- 协调通信组件：负责节点注册、消息封装、任务分发和回执汇总。

如果把整个系统看成一条处理链，那么这些组件之间的关系可以概括为：

1. 解析组件先把外部输入转换成系统可识别的数据对象。
2. 协调通信组件根据调度对象组织节点、封装消息并分发任务。
3. 执行组件在卫星节点侧接收任务并驱动行为树执行。
4. 条件计算组件和基础支撑组件为执行过程提供判断与公共能力。
5. 最终结果再通过协调通信组件回传到协调器侧。

### 4.1 `core` 模块

#### 4.1.1 `src/core/types.h`

该文件定义了系统中的基础数据结构：

- `NodeType`：行为树节点类型，包括 `SEQUENCE`、`SELECTOR`、`PARALLEL`、`ACTION`、`CONDITION`、`SUBTREE`。
- `BehaviorNode`：行为树节点定义，包含名称、类型、参数、表达式、命令、子节点、变量、常量和观测映射。
- `TaskSegment`：调度任务段，描述一条任务如何绑定到某颗卫星和某个行为定义。
- `CommandResult`：命令执行结果。
- `getCurrentTimeMs()`：获取当前毫秒级时间戳。

该文件相当于解析器、执行器和协调器之间共享的数据契约。

进一步看，`types.h` 在系统中承担的是“统一语义模型”的角色。也就是说，不同模块虽然职责不同，但它们对任务、行为树节点和命令输出的理解必须保持一致，而这一致性就是通过公共类型定义来保证的。

其中最关键的组件是 `BehaviorNode`。它不是简单的树节点结构，而是行为树执行时的完整语义载体，包含以下信息：

- 结构信息：`name`、`type`、`children`
- 控制信息：`expression`、`command`
- 数据信息：`params`、`variables`、`constants`、`variable_inits`
- 观测信息：`observation`

因此，`BehaviorNode` 同时承担了“控制流定义”和“运行时参数模板”两种职责。解析器负责构造它，执行器负责消费它。

`TaskSegment` 则是调度层和执行层之间的桥梁。它把调度文件中的一条任务映射为执行器能理解的最小执行单元，主要包含：

- 任务所属卫星：`satellite_id`
- 任务段标识：`segment_id`
- 任务编号：`task_id`
- 行为引用：`behavior_ref`
- 运行参数：`behavior_params`

可以把 `TaskSegment` 理解为“某颗卫星要以什么参数执行哪个行为模板”的描述对象。

从设计上看，这些基础类型全部放在 `core` 层，而不是散落在各业务模块中，有两个明显好处：

- 降低模块间耦合，避免解析器和执行器各自维护一套重复结构。
- 为后续增加新模块提供稳定的数据边界，例如测试模块、可视化模块或真实部署模块。

#### 4.1.2 `src/core/logger.h`

该文件实现轻量日志器 `Logger`，主要特点如下：

- 支持 `DEBUG`、`INFO`、`WARN`、`ERROR`、`OFF` 五种日志级别。
- 日志级别可通过环境变量 `MS_LOG_LEVEL` 控制。
- 日志输出自动附带时间戳和级别前缀。
- 通过宏 `LOG`、`LOG_DEBUG`、`LOG_WARN`、`LOG_ERROR` 简化调用。

日志模块被整个项目广泛依赖，是定位调度、通信和执行问题的主要手段。

从组件角度看，`Logger` 是一个全局单例型基础服务，所有模块都通过宏接口直接访问它。它的实现特点包括：

- 通过 `Logger::getInstance()` 提供全局唯一日志入口。
- 通过 `std::mutex` 保证多线程场景下输出不交叉。
- 在格式层统一补齐时间戳和日志级别，避免业务代码重复拼接。
- 在启动时读取环境变量 `MS_LOG_LEVEL`，实现运行级别配置。

这个设计虽然轻量，但非常符合当前工程规模。对于当前项目来说，日志器不需要复杂的异步写盘、日志轮转或多目标输出，重点是“稳定、统一、足够可读”。因此现有实现是一种典型的原型系统日志方案。

从调用方式看，代码中真正大量使用的是以下几个宏：

- `LOG`
- `LOG_DEBUG`
- `LOG_WARN`
- `LOG_ERROR`

这些宏屏蔽了日志对象访问细节，使业务代码可以把注意力集中在“记录什么”而不是“怎么记录”。这也是 `core` 模块作为基础支撑层的典型体现。

### 4.2 `parser` 模块

#### 4.2.1 `src/parser/json_parser.*`

该模块负责两类 JSON 数据解析。

第一类是调度文件解析：

- 类名：`ScheduleParser`
- 入口函数：`parseAllSatellites(const std::string& schedule_file)`
- 输出结构：`ScheduleParser::MultiSatSchedule`

解析结果包括：

- `satellite_ids`：卫星 ID 列表
- `satellites`：卫星基础信息列表
- `satellite_tasks`：每颗卫星对应的任务段数组

第二类是行为库解析：

- 类名：`BehaviorLibraryParser`
- 入口函数：`parseBehaviorDefinition(const std::string& library_file, const std::string& behavior_name)`

它会从 `behaviorTree.json` 中读取某个行为定义，并递归生成 `BehaviorNode` 树。

这个模块完成的是“把 JSON 变成内存对象”，还没有执行行为逻辑。

在 `parser` 模块内部，`json_parser` 实际上包含两个功能相近但职责不同的组件。

第一个组件是调度解析组件 `ScheduleParser`。它面向 `src/input/schedule.json`，主要解决的是“如何从输入文件中识别出系统中有哪些卫星、每颗卫星有哪些任务”。它完成的处理步骤包括：

1. 打开调度 JSON 文件。
2. 读取 `satellites` 数组。
3. 提取每颗卫星的 `satellite_id`、`node_id`、`name` 等信息。
4. 读取该卫星下的 `scheduled_tasks`。
5. 将每个任务项转换为 `TaskSegment`。
6. 最终汇总成 `MultiSatSchedule`。

这个组件的输出结果会直接被 `Coordinator` 消费，因此它相当于系统调度入口的第一道“数据整理层”。

第二个组件是行为库解析组件 `BehaviorLibraryParser`。它面向 `src/input/behaviorTree.json`，解决的问题是“根据一个行为名称，把对应的行为定义完整读取出来”。它的核心是递归函数 `parseNodeFromJson`，递归解析每个节点的：

- 类型
- 名称
- 参数
- 表达式
- 命令
- 变量定义
- 常量定义
- 观测映射
- 子节点

从实现特点上看，这个组件并不关心任务来自哪颗卫星，也不关心节点如何执行，它只负责把 JSON 描述忠实地恢复成 `BehaviorNode` 树。

因此，`json_parser` 模块整体可以概括为一句话：它负责把“外部配置语言”翻译成“内部对象模型”。

#### 4.2.2 `src/parser/behavior_parser.*`

该模块负责行为模板实例化，核心类是 `BehaviorTreeParser`。

入口函数：

- `instantiate(const BehaviorNode& definition, const std::map<std::string, std::string>& params)`

主要逻辑如下：

1. 合并模板变量默认值、常量和运行时参数。
2. 替换节点中的 `${var}` 占位符。
3. 递归展开整棵行为树。
4. 返回可直接执行的行为节点对象。

参数合并顺序很重要：

- 先加载变量默认值 `variable_inits`
- 再加载常量 `constants`
- 最后用任务运行时参数覆盖

因此，调度文件中的 `behavior_params` 拥有最高优先级。

如果说 `BehaviorLibraryParser` 负责“读取模板”，那么 `BehaviorTreeParser` 负责“实例化模板”。这是两个不同层次的工作。

行为树模板本身通常只描述通用逻辑，例如“移动到目标点”“搜索目标”“如果满足条件则上报结果”，但不会在模板里直接写死每次任务的具体坐标、目标名称和控制量。因此系统需要在运行时把任务参数注入模板，形成当前任务专属的行为树实例，而这正是 `BehaviorTreeParser` 的职责。

这个组件内部最关键的处理过程有两个：

1. 参数合并
2. 占位符替换

参数合并通过 `buildTemplateParams()` 完成，它会把变量初值、常量和运行时参数合成为一个统一参数表。占位符替换通过 `replaceVariables()` 完成，它会扫描文本中的 `${name}` 表达式，并用实际值替换。

需要特别注意的是，模板实例化不仅作用于 `params`，还会作用于：

- `command`
- `expression`
- 子节点中的对应字段

也就是说，这个组件的替换行为是递归传播到整棵树上的，而不是只替换根节点参数。

从系统协作上看，`BehaviorTreeParser` 是连接“行为模板定义”和“行为执行引擎”的中间层。如果没有它，执行器只能执行写死的树结构，无法支持面向任务的参数化行为复用。

### 4.3 `constraint` 模块

#### 4.3.1 `src/constraint/evaluator.*`

该模块实现 `ConstraintEvaluator`，用于判断行为树中的布尔条件是否成立。

它支持的能力包括：

- 比较运算：`==`、`!=`、`<`、`>`、`<=`、`>=`
- 逻辑运算：`&&`、`||`、`!`
- 括号表达式
- 布尔、数字、字符串和变量名混合比较

执行器在两个场景下会用到该模块：

- 节点存在 `expression` 前置条件时
- 条件节点 `Condition` 执行时

该模块只负责真假判断，不负责数值计算。数值表达式计算由执行器内部的 `NumericExpressionParser` 负责。

从实现方式上看，`ConstraintEvaluator` 是一个轻量级表达式求值组件，它没有依赖外部脚本引擎，而是自己完成了从字符串到布尔结果的解析流程。这个流程大致包括：

1. `tokenize()`：把字符串切分为数字、标识符、运算符、括号等标记。
2. `parseExpression()`：按逻辑优先级递归下降解析。
3. `parseOrExpression()` / `parseAndExpression()` / `parseNotExpression()`：处理逻辑运算。
4. `parsePrimary()`：处理变量、字面量和括号表达式。
5. `compare()`：对两个值进行最终比较。

这个组件的设计目标不是做通用脚本语言，而是为行为树执行提供“足够稳定的条件判断能力”。因此它具备以下特点：

- 功能范围可控，便于在 C++ 内部直接维护。
- 依赖 `VariableManager` 获取变量值，天然与执行器上下文绑定。
- 对布尔、数字、字符串三类值都提供了基本比较支持。

在行为树执行过程中，`ConstraintEvaluator` 有两个主要使用入口：

- 节点级前置条件判断
- 条件节点本体判断

也就是说，一个节点既可能因为前置条件不满足而被阻塞，也可能自己本身就是一个条件节点并返回真假。这个区分对于理解执行器逻辑很重要。

### 4.4 `executor` 模块

#### 4.4.1 `src/executor/variable_manager.*`

`VariableManager` 是执行期变量存储中心，负责维护三类变量作用域：

| 作用域 | 说明 |
| --- | --- |
| `GLOBAL` | 跨任务保留的全局状态 |
| `LOCAL` | 当前任务执行期的局部变量 |
| `INTERMEDIATE` | 中间输出变量，通常用于控制指令等临时结果 |

其核心能力包括：

- `set` / `get` / `exists`：读写变量
- `setFromParams`：把字符串参数自动转成 int、double、bool 或 string
- `loadFromScheduleConfig`：从调度文件读取某颗卫星的基础变量
- `createSnapshot` / `restoreSnapshot`：支持执行失败后的变量回滚

这是行为树能“带状态运行”的关键基础设施。

从组件内部结构看，`VariableManager` 由三部分能力组成：

1. 变量存取能力
2. 类型转换能力
3. 状态快照能力

变量存取能力由 `set()`、`get()`、`exists()` 和 `clearScope()` 构成，是最基础的数据访问层。类型转换能力由 `setFromParams()` 提供，它会把字符串尝试转换为整数、浮点数、布尔值，转换失败后再保留为字符串。这使得调度文件和行为树参数虽然以 JSON 或字符串形式进入系统，但进入执行期后可以以更自然的类型参与计算。

状态快照能力则是执行器容错机制的重要组成部分。每次执行任务时，执行器会先创建一个快照；如果任务执行失败且启用了回滚，变量管理器就会恢复到执行前状态。这个机制的意义在于：

- 避免失败节点把中间脏状态遗留到后续任务中。
- 为行为树执行提供最基本的事务性语义。

此外，`VariableManager` 还提供 `loadFromScheduleConfig()`，用于从调度文件加载某颗卫星的基础上下文。这说明它不仅是执行期局部变量容器，也承担了部分“节点基础状态初始化器”的职责。

在模块协作关系中，`VariableManager` 是 `GenericExecutor` 和 `ConstraintEvaluator` 的共同依赖对象。执行器写变量，条件求值器读变量，两者围绕同一个变量上下文工作。

#### 4.4.2 `src/executor/generic_executor.*`

`GenericExecutor` 是项目中最重要的执行引擎，负责真正跑行为树。其整体执行过程如下：

1. 创建任务执行上下文 `ExecutionContext`。
2. 初始化任务上下文变量，例如 `satellite_id`、`task_id`、`segment_id`。
3. 创建变量快照，用于失败回滚。
4. 调用 `BehaviorTreeParser` 完成行为模板实例化。
5. 初始化行为定义中的变量默认值和常量。
6. 递归执行行为树节点。
7. 如果某节点失败，则恢复快照。
8. 清理局部变量和中间变量。
9. 返回 `ExecutionResult`。

该执行器支持的节点语义如下：

| 节点类型 | 执行语义 |
| --- | --- |
| `ACTION` | 执行具体命令 |
| `CONDITION` | 计算条件真假 |
| `SEQUENCE` | 子节点按顺序执行，任一失败则失败 |
| `SELECTOR` | 子节点按顺序尝试，任一成功则成功 |
| `PARALLEL` | 当前实现是顺序遍历所有子节点，只要有失败即整体失败 |

需要注意，`PARALLEL` 在命名上是并行，但当前实现并没有真正开并发，而是顺序执行所有子节点后汇总结果。

从“组件内部结构”来看，`GenericExecutor` 可以拆成六个子职责：

1. 任务生命周期管理
2. 执行上下文管理
3. 变量初始化与回滚
4. 行为树递归调度
5. 表达式求值
6. 内置命令执行

任务生命周期管理主要由 `initialize()`、`shutdown()`、`executeTask()` 和 `workerLoop()` 构成。虽然当前卫星模拟器把执行器配置成串行执行，但执行器本身已经预留了异步队列和工作线程机制，说明它的设计目标并不只局限于当前样例，而是考虑过后续并发扩展。

执行上下文管理依赖 `ExecutionContext`。这个对象保存：

- 当前任务 ID
- 当前任务段 ID
- 任务开始时间
- 快照 ID
- 是否已取消

它的作用是把“当前这次任务执行”的运行信息集中管理，而不是散落在局部变量中。

变量初始化与回滚主要体现在以下几个步骤：

- `initializeTaskContext()` 写入任务级基础变量
- `initializeBehaviorDefaults()` 写入模板默认变量和常量
- `createSnapshot()` 创建回滚点
- 失败后调用 `restoreSnapshot()` 恢复状态

行为树递归调度由 `executeNode()` 及其派生函数组成。`executeNode()` 是执行器的核心分派函数，它完成以下工作：

1. 检查取消状态。
2. 应用观测值映射 `applyObservationValues()`。
3. 如果节点存在前置条件，则先做条件验证。
4. 根据节点类型分派到 `executeAction()`、`executeCondition()`、`executeSequence()`、`executeSelector()`、`executeParallel()`。
5. 汇总节点输出。

表达式求值又分为两层：

- 布尔约束调用 `ConstraintEvaluator`
- 数值表达式调用内部 `NumericExpressionParser`

这种拆分方式非常合理，因为布尔逻辑和算术逻辑虽然都属于“表达式”，但语义目标不同，分开实现更容易维护。

内置命令执行则是该组件与具体任务语义结合最紧密的部分。执行器本身并不知道“搜索目标”“上报目标”这些概念来自业务模型还是实验场景，它只是根据命令名执行相应变量操作或共享动作。因此，内置命令层可以理解为“行为树抽象语义”和“当前项目任务语义”之间的接口层。

#### 内置命令

`GenericExecutor::executeBuiltinCommand()` 当前实现了以下内置命令：

| 命令 | 作用 |
| --- | --- |
| `NO_OP` | 空操作 |
| `ASSIGN` | 计算表达式并写入局部变量 |
| `MOVE_TO` | 更新位置变量，并同步写入 `POS_X`、`POS_Y`、`POS_Z` |
| `SEND` | 把数据写入进程内共享邮箱 |
| `RECEIVE` | 从共享邮箱读取数据并回填局部变量 |
| `EMIT_COMMAND` | 输出控制类中间变量 |
| `NEXT_TASK` | 标记进入下一任务 |
| `REPORT_TARGET` | 写入目标上报结果 |
| `EXIT` | 标记任务退出 |
| `SETVAR ... = ...` | 直接修改全局变量 |

#### 表达式能力

执行器内部除了调用 `ConstraintEvaluator` 外，还自己实现了数值表达式求值器，用于支持：

- `+`
- `-`
- `*`
- `/`
- `sqrt(...)`

因此，像 `dist = sqrt(dx * dx + dy * dy + dz * dz)` 这样的表达式可以直接在行为树参数中使用。

为了更准确理解执行器，这里还需要补充说明几个关键内部机制。

第一，观测映射机制。`applyObservationValues()` 会根据节点的 `observation` 定义，把诸如 `POS_X`、`TARGET_X`、`REC` 这样的观测键值映射到本地变量中。这样一来，行为树模板就能用统一变量名编写，而不必直接依赖底层传感器字段名。

第二，输出汇总机制。节点执行完成后，执行器会把输出整理到 `ExecutionResult::outputs` 中，供上层模块打印日志或进一步封装结果。这使行为树执行不仅有“成功/失败”结果，还有一定程度的数据产出能力。

第三，局部作用域清理机制。每次任务结束后，执行器会清空 `LOCAL` 和 `INTERMEDIATE` 变量作用域，但保留 `GLOBAL`，这体现了一个很明确的状态边界：任务上下文可以结束，但卫星全局状态可以延续。

第四，运行时互斥机制。`executeTask()` 中使用 `runtime_mutex_` 包裹核心执行过程，说明当前执行器虽然有异步框架，但仍显式保护同一执行器实例上的运行时状态，避免多个任务同时读写同一套变量上下文。

#### 4.4.3 `src/executor/satellite_simulator.*`

`SatelliteSimulator` 可以理解为“单颗卫星在本进程中的代理节点”。它的职责包括：

- 初始化对应卫星的执行器。
- 从调度文件加载本卫星的基础变量。
- 接收协调器发来的批量任务消息。
- 为每个任务加载行为定义。
- 调用执行器逐条执行任务。
- 向协调器发送进度、完成和 ACK 回执。

需要注意的两个实现特点：

1. 卫星模拟器会缓存行为定义，避免同一 `behavior_ref` 被重复解析。
2. 执行器被配置为 `async_mode = false`，因此同一颗卫星当前按串行方式执行任务。

如果把系统按“协调端”和“节点端”分开理解，那么 `SatelliteSimulator` 就是节点端的门面组件。它把通信、行为加载和任务执行三个子过程封装在一个对象中，使协调器不需要了解卫星侧的执行细节。

具体来说，它内部可以拆成四个子职责：

- 通信接入：接收 `InterSatComm` 投递来的消息。
- 消息识别：判断消息类型并分派处理函数。
- 行为准备：根据 `behavior_ref` 加载或复用行为定义。
- 执行回传：调用执行器后发送 `TASK_PROGRESS`、`TASK_COMPLETE` 和 `BATCH_TASK_ASSIGN_ACK`。

在当前实现里，`SatelliteSimulator` 对外只暴露两个核心接口：

- `initialize()`：完成执行器和基础状态初始化
- `handleMessage()`：处理来自协调器的任务消息

其中真正的业务入口是 `handleBatchTaskAssign()`。它会顺序遍历批量任务，逐条转换为 `TaskSegment`，执行后再按结果更新 ACK。可以看出，这个组件不仅负责“执行”，还负责“把执行结果重新组织成通信协议对象”。因此它是执行层和通信层之间非常重要的适配器。

### 4.5 `coordinator` 模块

#### 4.5.1 `src/coordinator/message_types.h`

该文件定义通信层使用的主要协议对象，包括：

- `MessageType`：消息类型枚举
- `MessageHeader`：消息头
- `Message`：完整消息对象
- `TaskAssignMessage`、`BatchTaskAssignMessage`
- `BatchTaskAssignAck`
- `TaskProgressMessage`
- `TaskCompleteMessage`
- `NodeRegisterMessage`

它是通信模块、协调器和卫星模拟器之间的统一协议契约。

从组件设计角度看，`message_types.h` 不是一个普通的枚举集合，而是整个通信系统的协议定义中心。通信双方之所以能够解耦，就是因为它们都只依赖这份统一协议，而不直接依赖彼此的内部实现。

这个文件中的组件可以分成三层：

- 协议基础层：`MessageType`、`Priority`、`MessageHeader`、`Message`
- 节点管理层：`NodeRegisterMessage`、`NodeCapability`、`NodeStatus`
- 任务执行层：`TaskAssignMessage`、`BatchTaskAssignMessage`、`TaskProgressMessage`、`TaskCompleteMessage`、`BatchTaskAssignAck`

这种分层说明该协议并不只是为了传递任务数据，而是已经覆盖了节点注册、状态管理、任务控制和执行回执等多个维度，为后续扩展真实网络部署保留了协议基础。

#### 4.5.2 `src/coordinator/node_registry.*`

`NodeRegistry` 负责维护系统中的节点注册信息，主要完成以下工作：

- 注册卫星节点
- 注销节点
- 查询节点信息
- 维护节点别名到标准节点 ID 的映射

当前注册表更像一个轻量级内存索引，用来保证协调器知道“系统里有哪些卫星节点”和“它们的逻辑别名如何映射”。

更细致地看，`NodeRegistry` 内部维护了两套核心数据：

- `nodes_`：标准节点 ID 到节点信息的映射
- `alias_to_node_id_`：别名到标准节点 ID 的映射

这意味着它不仅能记录“节点有哪些”，还能处理“一个节点可能同时存在多个识别名”的问题。例如卫星 ID、通信节点 ID 和注册时传入的节点名，都可能指向同一个逻辑节点。

它的关键价值体现在两个方面：

- 降低协调器对节点标识细节的依赖
- 为后续从仿真节点平滑过渡到真实节点提供统一索引能力

在当前工程中，这个组件是轻量的，但它的结构设计是合理的，因为多节点系统中“节点标识统一管理”本身就是一个必须独立出来的问题。

#### 4.5.3 `src/coordinator/inter_sat_comm.*`

`InterSatComm` 是通信抽象层。虽然当前示例主要在单进程内运行，但这个模块已经具备较完整的通信框架：

- 消息头序列化与反序列化
- 优先级消息队列
- 本地消息处理器注册
- socket 初始化、监听、发送、接收
- 心跳线程和超时检查框架

当前项目默认的关键机制是：

- 如果目标节点已经注册了本地处理器，则消息直接投递给对应处理函数。
- 如果没有本地处理器，消息才进入发送队列，等待真实网络链路处理。

这使得项目既能保留“像分布式系统一样的通信抽象”，又能以较低复杂度在单进程中演示完整流程。

从组件视角分析，`InterSatComm` 是当前系统中抽象程度最高的模块之一。它试图用统一接口同时覆盖：

- 本地进程内消息投递
- socket 数据收发
- 发送与接收队列管理
- 协议序列化与反序列化
- 心跳与节点连接维护

因此，这个模块内部其实包含多个子组件：

1. `MessageQueue`
2. `MessageSerializer`
3. 本地处理器注册表
4. 远端节点表
5. 多个后台线程框架

其中，`MessageQueue` 提供基于优先级的消息缓冲能力；`MessageSerializer` 负责把 `Message` 和批量任务对象转换成二进制或 JSON 载荷；本地处理器注册表用于实现单进程直投；远端节点表用于维护网络节点连接状态。

从当前项目的运行方式来看，最常用的其实是“本地直投路径”，即：

1. 协调器调用 `sendMessage()`
2. 通信模块发现目标节点已注册本地 handler
3. 直接调用对应处理函数

这条路径绕开了真实网络，但不绕开通信抽象，因此非常适合当前仿真工程。

如果未来要做真实多进程或跨主机扩展，最值得复用的也正是这个组件，因为它已经把消息封装、队列管理和连接管理的边界预先整理出来了。

#### 4.5.4 `src/coordinator/coordinator.*`

`Coordinator` 是项目的总控中心，承担以下职责：

- 加载调度文件
- 初始化节点注册表与通信模块
- 创建卫星模拟器
- 向各卫星分发批量任务
- 接收并统计 `ACK`、进度、完成消息
- 在超时后给未完成卫星判失败

其中最重要的几个函数如下：

| 函数 | 作用 |
| --- | --- |
| `initialize()` | 完成系统初始化 |
| `run()` | 启动通信并分发任务，等待执行结果 |
| `registerSatelliteNodes()` | 按调度文件注册卫星节点 |
| `distributeAllTasks()` | 将每颗卫星的任务打包为 `BATCH_TASK_ASSIGN` |
| `initializeSimulators()` | 创建并初始化卫星模拟器 |
| `onCoordinatorBatchAck()` | 处理卫星批次回执 |

从职责划分上看，`Coordinator` 负责“组织和调度”，不直接负责“任务逻辑执行”。这符合模块边界设计。

在整个系统里，`Coordinator` 是唯一的顶层编排组件。它不解析行为细节，不执行任务逻辑，也不维护复杂表达式，但它负责把所有子组件组织成一个完整系统。

从内部结构看，`Coordinator` 主要管理以下几类状态：

- 系统配置：`config_`
- 通信组件：`comm_`
- 节点注册表：`node_registry_`
- 调度对象：`schedule_`
- 卫星模拟器表：`simulators_`
- ACK 与任务完成跟踪表：`satellite_task_results_`、`satellite_task_finished_`、`satellite_task_expected_counts_`
- 消息路由表：`coordinator_message_router_`

这说明它并不是一个简单的控制器，而是一个系统级状态管理中心。

从方法职责上，可以把 `Coordinator` 分为四个子过程：

1. 初始化过程
2. 任务分发过程
3. 消息接收与路由过程
4. 结果汇总与关闭过程

初始化过程负责搭好系统骨架；任务分发过程负责把 `schedule_` 中的数据转为批量任务消息；消息接收与路由过程负责把不同消息类型分派到对应处理函数；结果汇总过程则负责基于 ACK 统计各卫星任务是否成功完成。

需要特别指出的是，当前协调器虽然注册了 `TASK_PROGRESS` 和 `TASK_COMPLETE` 的处理入口，但其核心结束条件仍然依赖 `BATCH_TASK_ASSIGN_ACK`。这意味着当前实现更偏向“批次完成判断”而不是“精细化任务监控”。这也是后续可以增强的一个方向。

## 五、关键数据流说明

### 5.1 调度数据流

调度数据从 `src/input/schedule.json` 进入系统，经过以下路径流动：

1. `ScheduleParser` 解析 JSON。
2. 生成 `MultiSatSchedule`。
3. `Coordinator` 按卫星分组读取任务。
4. 转换为 `BatchTaskAssignMessage`。
5. 通过 `InterSatComm` 投递给对应 `SatelliteSimulator`。

### 5.2 行为数据流

行为数据从 `src/input/behaviorTree.json` 进入系统，经过以下路径流动：

1. `BehaviorLibraryParser` 根据 `behavior_ref` 读取行为定义。
2. `BehaviorTreeParser` 将模板参数替换成当前任务的运行参数。
3. `GenericExecutor` 递归执行实例化后的行为树。

### 5.3 状态数据流

执行状态主要保存在 `VariableManager` 中，典型流向如下：

1. 卫星基础状态从调度文件载入 `GLOBAL`。
2. 任务运行参数进入 `LOCAL`。
3. 命令输出和中间控制量写入 `INTERMEDIATE`。
4. 执行完成后清空 `LOCAL` 和 `INTERMEDIATE`，保留 `GLOBAL`。

### 5.4 回执数据流

卫星执行任务后，会向协调器发送三类信息：

- `TASK_PROGRESS`：任务执行中间状态
- `TASK_COMPLETE`：单任务完成情况
- `BATCH_TASK_ASSIGN_ACK`：整批任务总体结果

当前协调器最终用于结束等待的是 `BATCH_TASK_ASSIGN_ACK`，也就是“批次级结果”。

## 六、示例输入文件说明

### 6.1 `src/input/schedule.json`

该文件定义示例调度计划。当前默认包含 6 颗卫星：

- `S1` 到 `S5`：每颗执行 `GoToStation` 和 `Search` 两个任务
- `S6`：只执行 `Search`

每个任务段最关键的字段是：

- `segment_id`：任务段唯一标识
- `task_id`：任务编号
- `behavior_ref`：引用的行为定义名称
- `behavior_params`：运行时参数

### 6.2 `src/input/behaviorTree.json`

该文件定义行为树库。当前主要包含两个示例行为：

1. `GoToStation`
2. `Search`

其中：

- `GoToStation` 主要演示位置观测、机动、到位判断和控制输出。
- `Search` 主要演示观测、条件判断、距离计算、星间共享、目标确认和最终上报。

这两个示例已经覆盖了顺序、选择、条件、赋值、共享通信、位置更新等主要机制，适合作为阅读整个执行引擎的入口。
